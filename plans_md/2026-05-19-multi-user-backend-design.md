# Multi-user backend — design

**Date**: 2026-05-19
**Status**: design approved, no implementation started
**Author**: Max + Claude (brainstorming session)
**Next step**: write implementation plan (`writing-plans` skill)

A design for migrating the dashboard from single-tenant JSON-file storage to a multi-user PostgreSQL-backed system with invite-only signup and per-user data isolation.

---

## Goal and non-goals

**Goal**: take the current single-user dashboard and turn it into a small-community multi-user app. Each user logs in, sees only their data, manages their own integration credentials (Steam, Canvas, GitHub). The existing single user (Max) becomes `user_id = 1` with all current data preserved.

**Non-goals**:
- Password reset via email — separate spec when email-sending is set up
- OAuth providers — separate spec
- Multi-tenant data sharing ("share my plan with my partner") — current design is strict per-user isolation
- Email notifications of any kind
- Cross-user features (shared lists, social anything)
- Migrating the tools page (it is 100% stateless and stays shared)
- Migrating the news/favicon endpoints (they are public RSS proxies and stay shared)

## Audience and scale

- **Audience**: small community, ~10–100 users, invite-only
- **Hosting**: same single Hetzner VPS the app runs on today
- **Concurrent writers**: low (each user editing their own todos)
- **Data volume**: ~5 MB total at 100 users at current data shapes
- **Implementation pace**: evenings, ~12 evenings of focused work spread over 2-3 weeks

## Architecture

```
┌─────────────────────────────────────────────────────┐
│  Browser                                            │
│  - sets session cookie (httpOnly, secure)           │
└─────────────────────────────────────────────────────┘
                       │
                       ▼ (HTTPS via nginx)
┌─────────────────────────────────────────────────────┐
│  /opt/dashboard/www/api.py  (ThreadingHTTPServer)   │
│  Same single Python service, just multi-user:       │
│  - cookie → session lookup → user_id                │
│  - every data endpoint scopes WHERE user_id = ?     │
│  - new endpoints: /api/auth/{login,signup,logout}   │
└─────────────────────────────────────────────────────┘
            │                              │
            ▼                              ▼
┌──────────────────────┐      ┌──────────────────────┐
│  PostgreSQL 16        │      │  tools_api.py port  │
│  systemd service      │      │  5002 — unchanged   │
│  localhost:5432       │      │  (stateless tools)  │
└──────────────────────┘      └──────────────────────┘
```

Key decisions:
- One Python service stays the source of truth. No new framework. Existing `ThreadingHTTPServer` keeps working; we add a session-loading middleware.
- Postgres on `localhost:5432`, bound to localhost only. `pg_dump` cron to `/opt/dashboard/backups/`.
- `notes_api.py` (port 5001) folds into `api.py` during the migration. No reason to keep two Python services once we have a real DB. `tools_api.py` (port 5002) stays separate because it has different timeout/upload-size needs and zero per-user state.
- Sessions are server-side (not JWT). A `sessions` table maps cookie token → user_id. Logout = delete row. Lets us force-logout users by revoking session rows.
- Per-user secrets (Steam/Canvas tokens) encrypted at rest in the user_integrations table, using AES-GCM with a key in `/etc/dashboard.env`. Compromised DB dump alone can't read the tokens; an attacker would need both DB + env file.

## Database schema

Postgres 16. Two flavors: relational tables for boring data, JSONB columns for flexible blobs (home layout, per-user caches).

```sql
-- Requires: CREATE EXTENSION citext;

-- Identity and auth
CREATE TABLE users (
  id            BIGSERIAL PRIMARY KEY,
  email         CITEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL,            -- argon2id
  display_name  TEXT NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE invite_codes (
  code            TEXT PRIMARY KEY,        -- random 16-char URL-safe
  created_by_id   BIGINT REFERENCES users(id),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  used_by_id      BIGINT REFERENCES users(id),
  used_at         TIMESTAMPTZ
);
-- Single-use only. If you ever want multi-use codes, add a separate
-- invite_redemptions table rather than counters on this row.

CREATE TABLE sessions (
  id          TEXT PRIMARY KEY,            -- 32-byte URL-safe random token
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at  TIMESTAMPTZ NOT NULL,
  last_seen   TIMESTAMPTZ NOT NULL DEFAULT now(),
  user_agent  TEXT
);
CREATE INDEX ON sessions (user_id);
CREATE INDEX ON sessions (expires_at);

-- Per-user integrations (Steam, Canvas, GitHub) — encrypted at rest
CREATE TABLE user_integrations (
  user_id          BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  integration_type TEXT NOT NULL,          -- 'steam' | 'canvas' | 'github'
  payload_enc      BYTEA NOT NULL,         -- AES-GCM ciphertext of JSON config
  iv               BYTEA NOT NULL,         -- 12-byte nonce
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, integration_type)
);

-- Boring relational tables
CREATE TABLE todos (
  id           BIGSERIAL PRIMARY KEY,
  user_id      BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  text         TEXT NOT NULL,
  priority     TEXT NOT NULL CHECK (priority IN ('high','medium','low')),
  deadline     DATE,
  done         BOOLEAN NOT NULL DEFAULT FALSE,
  pinned       BOOLEAN NOT NULL DEFAULT FALSE,
  completed_at TIMESTAMPTZ,
  position     INT NOT NULL DEFAULT 0,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON todos (user_id, done, position);

CREATE TABLE plan_events (
  id          BIGSERIAL PRIMARY KEY,
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title       TEXT NOT NULL,
  date        DATE NOT NULL,
  start_time  TIME,
  end_time    TIME,
  tag         TEXT,
  location    TEXT,
  color       TEXT,
  recurring   BOOLEAN NOT NULL DEFAULT FALSE
);
CREATE INDEX ON plan_events (user_id, date);

CREATE TABLE notes (
  id          BIGSERIAL PRIMARY KEY,
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title       TEXT NOT NULL,
  body        TEXT NOT NULL DEFAULT '',
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON notes (user_id, updated_at DESC);

CREATE TABLE categories (
  id          BIGSERIAL PRIMARY KEY,
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,
  position    INT NOT NULL DEFAULT 0,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON categories (user_id, position);

CREATE TABLE links (
  id          BIGSERIAL PRIMARY KEY,
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  category_id BIGINT REFERENCES categories(id) ON DELETE SET NULL,
  url         TEXT NOT NULL,
  name        TEXT NOT NULL,
  sub         TEXT,
  color       TEXT,
  icon_type   TEXT,                       -- 'favicon' | 'svg' | 'image' | 'emoji'
  icon_value  TEXT,
  favorite    BOOLEAN NOT NULL DEFAULT FALSE,
  position    INT NOT NULL DEFAULT 0,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON links (user_id, category_id, position);

-- Flexible / blob-shaped data
CREATE TABLE home_layout (
  user_id    BIGINT PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  payload    JSONB NOT NULL DEFAULT '{}'::jsonb,
  -- payload = { version, sections[], widgets[], habits[] }
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE cache_skole (
  user_id     BIGINT PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  payload     JSONB NOT NULL,
  fetched_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE cache_wishlist (
  user_id     BIGINT PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  payload     JSONB NOT NULL,
  fetched_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Global reports (any logged-in user can submit; admin sees all)
CREATE TABLE reports (
  id          BIGSERIAL PRIMARY KEY,
  user_id     BIGINT REFERENCES users(id) ON DELETE SET NULL,  -- nullable for anon
  type        TEXT NOT NULL CHECK (type IN ('bug','feature')),
  title       TEXT NOT NULL,
  body        TEXT,
  page        TEXT,
  status      TEXT NOT NULL DEFAULT 'open',
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Schema notes:
- `CITEXT` makes email case-insensitive. Requires `CREATE EXTENSION citext` in the initial migration.
- Every per-user table has `ON DELETE CASCADE` so deleting a user wipes their data cleanly.
- `position INT` on orderable lists (todos, links, categories). Drag-reorder commits write a batch UPDATE. Could later swap for `lexorank` strings if reordering becomes hot.
- `home_layout` is one row per user with a JSONB payload that directly mirrors the current `home.json` shape so the frontend hooks (`useHome`, `useWidgets`, `useHabits`) don't change shape.
- `user_integrations.payload_enc` uses raw BYTEA (not pgcrypto) because pgcrypto column-encryption needs the DB process to hold the key, breaking the "DB dump alone is useless" property. AES-GCM in Python with the key from `/etc/dashboard.env` is simpler and gives a stronger threat model.
- The encrypted `payload` (plaintext form) is a JSON object whose shape varies by integration type. Example: `{"api_key": "...", "steam_id": "..."}` for steam, `{"token": "...", "base_url": "https://hiof.instructure.com/api/v1", "courses": [10644, 10666]}` for canvas, `{"username": "Maxaubert", "repo": "PDP2026/labs-Maxaubert"}` for github. The `iv` is a fresh 12-byte random nonce per write.
- `reports.user_id` is nullable so reports submitted before authentication still attach. Long-term reports stay a global feedback inbox visible to admin.

## Auth flow

**Signup** — `POST /api/auth/signup`
1. Body: `{ code, email, password, display_name }`
2. Server validates: invite code exists AND `used_at IS NULL`. Invalid → 401.
3. Hash password with `argon2id` (time_cost=3, memory_cost=65536, parallelism=4).
4. Insert user, mark invite code consumed (`UPDATE invite_codes SET used_by_id = $1, used_at = now() WHERE code = $2 AND used_at IS NULL` — the `WHERE used_at IS NULL` guards against a race where two requests claim the same code; whichever wins the UPDATE proceeds, the other gets 0 rows affected and 401), create session row, set cookie.
5. Response: `{ user: { id, email, display_name } }` plus `Set-Cookie: session=<token>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000` (30 days).

**Login** — `POST /api/auth/login`
1. Body: `{ email, password }`
2. Look up by `email` (CITEXT), verify with `argon2.verify(hash, password)`. Constant-time. Wrong creds → 401.
3. Same session creation + cookie as signup.

**Session middleware** — runs at the top of every `do_GET` / `do_POST`
- Read `session` cookie. Empty → `current_user = None`.
- Look up session by id. Expired or not found → clear cookie, `current_user = None`.
- Otherwise update `last_seen`, attach `current_user` to request context.
- Endpoints requiring auth: `if current_user is None: 401`.
- Endpoints scope data: `WHERE user_id = current_user.id`.

**Logout** — `POST /api/auth/logout` — delete session row, clear cookie, 204.

**Session cleanup** — a daily cron (or a postgres `pg_cron` job if installed) runs `DELETE FROM sessions WHERE expires_at < now()`. Without it the sessions table grows indefinitely; with it, expired rows go away within 24h of expiry. Stale sessions still get rejected at request time by the middleware regardless.

**CSRF**: `SameSite=Lax` covers cross-site GET/POST attacks. Single dashboard origin, no third-party requests accepted. No double-submit token needed.

**Rate limiting** (in-memory, per-IP, per-route):
- Login: 5 failures / 15 min / IP → lockout 15 min. Reset on successful login.
- Signup: 10 attempts / hour / IP. Stops invite-code brute force.
- Reports: 30 / hour / user.

**Password resets**: not in MVP. Workaround: admin issues a new invite or runs `UPDATE users SET password_hash = ...` via psql.

**Admin** — gated by `user_id = 1` for v1 (no separate role column). Routes:
- `POST /api/admin/invites` — create N invite codes
- `GET /api/admin/users` — list users
- `GET /api/admin/reports` — list all reports across users

## API mapping

| Endpoint | Auth | Behavior change |
|---|---|---|
| `GET /api/todos` | required | `SELECT * FROM todos WHERE user_id = $1 ORDER BY position` |
| `POST /api/todos` | required | bulk upsert with `user_id = $1`, re-renumber positions |
| `GET /api/plan` | required | `WHERE user_id = $1 ORDER BY date, start_time` |
| `POST /api/plan` | required | bulk upsert |
| `GET /api/notes` | required | `WHERE user_id = $1 ORDER BY updated_at DESC` |
| `POST /api/notes` | required | insert new |
| `PUT /api/notes/:id` | required | `WHERE id = $1 AND user_id = $2` (ownership check) |
| `DELETE /api/notes/:id` | required | `WHERE id = $1 AND user_id = $2` |
| `GET /api/links` | required | join links + categories, scoped |
| `POST /api/links` | required | bulk upsert (categories + links together) |
| `GET /api/home` | required | `SELECT payload FROM home_layout WHERE user_id = $1` |
| `POST /api/home` | required | `INSERT … ON CONFLICT (user_id) DO UPDATE SET payload = $1` |
| `GET /api/skole` | required | check cache, refetch via decrypted Canvas token, update cache |
| `GET /api/wishlist` | required | same shape using decrypted Steam config |
| `GET /api/pdf?stat=…` | required | uses user's Canvas token; per-user disk cache |
| `GET /api/news?source=…` | public | unchanged |
| `GET /api/favicon?domain=…` | public | unchanged |
| `POST /api/report` | optional | set `user_id` if logged in, else null |
| `GET /api/admin/reports` | admin | global list |
| `POST /api/auth/{signup,login,logout}` | various | new |
| `GET /api/auth/me` | public | returns current_user or 401 |
| `POST /api/admin/invites`, `GET /api/admin/users` | admin | new |

**API style**: keep the current "POST the whole list" pattern for todos/plan/links rather than switching to fine-grained REST. Frontend hooks don't change shape. The server-side upsert is `INSERT … ON CONFLICT (id) DO UPDATE` followed by `DELETE WHERE user_id = $1 AND id NOT IN (...)`.

**Connection pooling**: `psycopg.pool.ConnectionPool(min_size=2, max_size=10)` at module scope. Each request thread checks out via `with pool.connection() as conn:`. Localhost Postgres handles 10 connections trivially.

## Migration plan — existing data → user_id = 1

One-shot script `_migrate_to_postgres.py` (gitignored, like `_setup_env_secrets.py` was):

1. Connect via SFTP to the VPS using `dashboard.txt` credentials.
2. Read JSON files: `/opt/dashboard/www/{todos,plan,links,home}.json` + `/opt/dashboard/notes_data.json`.
3. Create `user_id = 1`, email prompted (default Max's), display_name = "Max". Prompt for a password.
4. Read existing `STEAM_API_KEY`, `STEAM_ID`, `CANVAS_TOKEN` from `/etc/dashboard.env`, encrypt with AES-GCM, insert as `user_integrations` rows (`integration_type = 'steam' | 'canvas'`).
5. Insert per-table data preserving client-side IDs where present (`todos.id` is a string in JSON; cast to BIGINT during insert; keep an `external_id` column initially if needed for frontend reconciliation, then drop it once verified).
6. Verify counts: `count(todos in JSON) == SELECT COUNT(*) FROM todos`, etc. Print diff.
7. Back up JSON to `/opt/dashboard/backups/migration-<date>/` before any deletion.
8. Don't delete `/opt/dashboard/www/*.json` automatically. That's a separate manual step after the new system runs successfully for ~1 week.

Reports markdown stays as-is — append-only, not migrated.

## Implementation order

Each item is its own PR cycle. Items 1–3 are critical path; 4–7 can land incrementally.

```
[ ] 1. Postgres install + schema migration tooling     ~1 evening
       apt install postgresql-16, localhost-only bind
       psycopg + yoyo-migrations added to requirements
       001_initial.sql applies the full schema
       server/db.py wraps the connection pool
       This PR adds zero behavior change to existing endpoints

[ ] 2. Auth endpoints + session middleware             ~2 evenings
       /api/auth/{signup,login,logout,me}
       server/auth.py: require_auth, current_user_from_cookie
       argon2id password hashing
       Rate-limiting helper
       Admin invite endpoint (admin = user_id = 1)
       All existing endpoints stay open / single-user during this phase
       Tests: signup → login → me → logout flow

[ ] 3. Migration script + first scoped endpoint        ~2 evenings
       _migrate_to_postgres.py converts JSON → rows
       Migrate /api/todos first as the pattern
       Frontend stays unaware (response shape unchanged)
       Verify: log in, see your todos, edit one, restart, edit persists
       Pattern validated; next sub-project is mechanical

[ ] 4. Migrate remaining data endpoints                ~2-3 evenings
       /api/plan, /api/links, /api/notes, /api/home one at a time
       Each gets its own commit so issues are isolatable
       Fold notes_api.py port 5001 into api.py here

[ ] 5. Migrate cached/integration endpoints            ~1 evening
       /api/skole, /api/wishlist, /api/pdf
       Pull secrets from user_integrations (decrypted)
       AES-GCM helper in server/crypto.py with key from /etc/dashboard.env

[ ] 6. Frontend auth + AccountSettings                 ~2-3 evenings
       LoginPage.tsx, SignupPage.tsx
       useCurrentUser hook gating App.tsx
       AccountSettings.tsx for display name + integration tokens
       401 handling in useToast / error boundaries

[ ] 7. Cleanup                                         ~1 evening
       Delete /opt/dashboard/www/*.json after backup
       Remove the migration script
       Update CLAUDE.md with auth + DB conventions
       Update systemd unit if anything changed
```

Total estimate: ~12 evenings, ~2-3 weeks part-time.

**Roll-forward safety**:
- Steps 1, 2, 5, 6 are additive — don't touch existing JSON-file behavior.
- Step 3 is the first risky cutover but only one endpoint. Revert one commit and you're back on JSON.
- Step 4 is incremental; each endpoint is independent. You can pause partway and ship.
- Step 7 is the destructive step. Don't run until the new system has been used for ≥1 week.

## Locked-in decisions (recap)

| Area | Decision |
|---|---|
| Storage | PostgreSQL 16 on the same VPS, systemd service, `localhost:5432` only |
| Auth | Invite codes + email/password (argon2id) |
| Sessions | Server-side, cookie holds session id, 30-day TTL |
| Schema | Pragmatic: relational tables + JSONB for home_layout and per-user caches |
| Per-user secrets | App-layer AES-GCM encryption, key in `/etc/dashboard.env` |
| API style | Keep current "bulk POST whole list" pattern; new auth endpoints under `/api/auth/*` |
| Reports | Stay global, `user_id` nullable for anon submissions |
| Frontend | Login/Signup pages + `useCurrentUser` gate + `AccountSettings` for integrations |
| Admin | `user_id = 1` checks for v1 (no role column yet) |
| Rate limiting | In-memory, per-IP, per-route |
| Migration tool | yoyo-migrations (lightweight) |
| Python DB client | psycopg 3 (with `psycopg.pool.ConnectionPool`) |
| Notes service | Folded into `api.py` during step 4 |
| Tools service | Stays separate (port 5002) |

## Open questions / future work

These are deliberately out of MVP scope and become their own specs later if needed:

- **Password reset via email** — needs SMTP provider (Resend / Postmark / SES). Add when first user actually loses a password.
- **OAuth (GitHub / Google sign-in)** — alternative or additional signup path.
- **Multi-tenant data sharing** — "share my plan with my partner" / shared lists.
- **Cross-device localStorage sync** — move UI prefs (`todo-view`, `news-source`, `weather-location-v1`) server-side per-user so they sync across browsers.
- **Real role-based admin** — `users.role` column with finer permissions than "is `id == 1`".
- **Email notifications** — daily digest, reminders, etc.
- **Tools-page per-user storage** — if any tool grows to need server-side state (saved QR codes, conversion history, etc.).
- **Audit log** — track who changed what, especially admin actions.
